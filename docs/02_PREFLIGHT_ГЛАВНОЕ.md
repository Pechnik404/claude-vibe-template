# PREFLIGHT — главное архитектурное решение вашей боли

> Это самый важный документ инфраструктуры. Прочитать обязательно.

---

## Ваша боль (ваша же формулировка)

> AI не всегда проверяет достоверность решения / тех.документации через Context7 и мировые стандарты, всегда поверхностно знакомится с доками, кодом — типа экономия токенов — приводит к дрейфу. На сэкономленные 10к токенов — потеря 500к токенов дрейфом, а потом фраза «ой, извини». Наглядно: Claude Opus 4.7 не знал про Claude Design и делал фронтэнд сам — минус месяц лимитов за ночь.

## Почему это происходит (Дост. 90%)

1. **Cutoff модели**. У любого LLM есть дата обрезки знаний. Anthropic выпускает новые продукты после cutoff — модель про них не знает и не помнит.
2. **«Знание» = вероятностный паттерн**. AI может «вспомнить» API библиотеки по обрывкам тренировочных данных. Это часто близко к правде, но не идентично. Особенно — устаревшие версии.
3. **Стимул экономить токены**. Длинный preflight = много токенов = «нерентабельно». LLM по умолчанию идёт по короткому пути.
4. **Признанная отраслевая проблема**. Martin Fowler — «premature implementation». OWASP — отдельный класс багов из-за устаревших ассумпций об API.

## Доказательство того, что Anthropic тоже видит проблему

Anthropic в 2026 выпустила **отдельный Frontend Design plugin** для Claude Code. Причина — ровно то, что вы описали: Claude в чате склонен изобретать UI с нуля, игнорируя готовый Claude Design. Плагин буквально решает один частный случай этой большой проблемы.

[Frontend Design – Claude Plugin](https://claude.com/plugins/frontend-design) — ставится в `.claude/plugins/`.

## Решение в нашей инфраструктуре

Не **уговоры** AI «пожалуйста, делай research», а **архитектурный замок**: невозможно начать писать код без свежей записи в `RESEARCH_LOG.md`.

### Три уровня защиты

**Уровень 1: правило в `CLAUDE.md` §1**
Чтение конституции при каждой сессии. AI знает, что preflight обязателен.

**Уровень 2: skill `preflight-research`**
Чёткий алгоритм 5 шагов: prior art → docs via Context7 → frontend design check → запись в RESEARCH_LOG → решение. Модель имеет инструкцию КАК делать research, не должна изобретать.

**Уровень 3: hook `preflight_gate.py` (главный замок)**
Перехватывает попытку Write/Edit любого файла > 50 строк. Проверяет дату последней записи в RESEARCH_LOG.md. Если > 60 минут назад или вообще нет — **БЛОК**. Сообщение пользователю + AI: «нужен preflight».

Это не уговор. Это `exit 2` из hook → Claude Code физически не даст продолжить.

## Скорость преимуществ

Один preflight занимает **5–15 минут** (поиск + загрузка доков + запись).
Один эпизод дрейфа стоит **часы или дни** (исправление + потерянные токены + потерянное настроение).

**ROI**: 5–15 мин против дней. Экономика очевидна.

## Что preflight конкретно делает

### Шаг 1. Find prior art (поиск готовых решений)

WebSearch + GitHub topic search. Фильтр: star > 500, last-commit < 6 мес.
Топ-5 решений с вердиктами: ✅ форк / ✅ библиотека / ⚠️ вдохновляемся / ❌ не подходит.

**Чем закрывает боль**: не пишем велосипед, если готовое работает.

### Шаг 2. Context7 deep-dive

Для каждой используемой библиотеки:
- `resolve-library-id` → находим точное имя в реестре доков
- `query-docs` → загружаем АКТУАЛЬНУЮ документацию в контекст

**Чем закрывает боль**: AI не «вспоминает» API, а читает свежие доки.

### Шаг 3. Frontend Design check

Если задача про UI:
- Проверяем наличие Frontend Design plugin
- Если нет — устанавливаем
- WebSearch по shadcn / HTMX patterns / Tailwind UI

**Чем закрывает боль**: ровно ваш кейс с Claude Design.

### Шаг 4. Запись в RESEARCH_LOG.md

Формализованный шаблон. Дата, что искал, что нашёл, что выбрал, confidence.

**Чем закрывает боль**: следующий раз AI не повторяет research — читает запись. Накопленный research = ваш ratchet (см. комбо-приём H).

### Шаг 5. Гейт

Hook проверяет наличие записи. Если нет — блок.

**Чем закрывает боль**: невозможность пропустить research даже «случайно».

## Метрики (отслеживать в LAB_JOURNAL)

- Кол-во preflight-проверок в неделю
- Среднее время preflight
- Сколько раз preflight нашёл готовое решение → сэкономил часы
- Сколько раз preflight выявил устаревшее API → предотвратил дрейф
- Кол-во срабатываний hard-gate (полезно знать — много = AI ещё привыкает)

## Что вы должны делать (минимум)

- При новой идее — дать AI несколько минут на preflight (не торопить)
- Иногда смотреть `RESEARCH_LOG.md` — убедиться что research реальный
- Если AI пытается обойти preflight — указать на это

## Что вы можете НЕ делать

- Самим запускать preflight — AI делает сам по правилам
- Самим следить за hooks — они работают автоматически
- Контролировать каждую запись — у вас доверие, у AI журналирование

## Если AI всё-таки начал дрейфить

1. СТОП: «AI, остановись. Покажи RESEARCH_LOG за эту задачу.»
2. Если записи нет — preflight был пропущен, hook сломан или AI обошёл. Расследуем.
3. Если запись есть, но дрейф случился — research был поверхностный. Обновляем skill `preflight-research`.

В обоих случаях — запись в `LAB_JOURNAL.md` с причиной, чтобы не повторить.

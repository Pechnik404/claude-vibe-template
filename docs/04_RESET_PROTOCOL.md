# RESET PROTOCOL — чистый перезапуск застрявших проектов

> Под ваши 2 застрявших проекта: сайт и тот, что застрял на VDS Selectel.
>
> Это **не позор** — это здоровое решение, когда тех.долг превысил критическую массу.

---

## Когда применять reset

✅ Применять:
- Дрейф контекста — 5+ раз падал и переписывался
- Невозможно сказать «как точно должно работать» (нет spec)
- Среда разработки не соответствует production (VDS-мазохизм)
- Стек/архитектура выбраны неудачно с самого начала
- Эмоциональная усталость + объективные баги

❌ НЕ применять:
- Локальная проблема (1-2 файла) — лучше точечный рефакторинг
- Тестов нет, но архитектура здоровая — добавьте тесты
- Только эмоциональная усталость — отдохните день, продолжите

## Шаги протокола

### Шаг 1. POST_MORTEM (без обвинений, без эмоций)

Создать `legacy/POST_MORTEM_<date>.md`:

```markdown
# Post-mortem: <название проекта>
Дата решения reset: 2026-XX-XX

## Цель проекта (что задумывалось)
<2-3 абзаца — как должно было работать>

## Что РАБОТАЛО
- <модуль А> — стабильно
- <функция Б> — частично, с ограничением

## Что СЛОМАЛОСЬ или НЕ дошло до реализации
- <фича X> — причина: <дрейф / устаревшее API / VDS-мазохизм>
- <фича Y> — причина: ...

## КОРНЕВЫЕ причины (типизация)
☐ Отсутствие spec
☐ Отсутствие тестов
☐ Выбор не той библиотеки
☐ Несоответствие среды разработки и production ⭐ (про VDS-проект)
☐ Контекстный дрейф AI без preflight ⭐ (про оба)
☐ Преждевременная оптимизация
☐ Иное: <описать>

## Уроки (одно предложение каждый)
1. ...
2. ...
3. ...
```

### Шаг 2. SALVAGE (что переносим)

Создать `legacy/SALVAGE_<date>.md`:

```markdown
## Переносим в новый репо
- `path/old/working_module.py` → `src/new/<имя>.py` — работало стабильно
- Схема БД (структура без данных) — переиспользуем

## Архивируем для истории
- Старый репо целиком — branch `legacy/final-state`
- README старого репо — обновить: «Перезапущен в <ссылка>»

## Выбрасываем
- `path/broken_module.py` — было причиной дрейфа
- node_modules (если был), кеши, .env
```

### Шаг 3. SPEC новой системы

Создать `docs/SPECS/<feature>.md` для каждой основной фичи:

```markdown
## Цель
<одно предложение>

## Сценарии использования
1. Пользователь делает X → система Y → результат Z
2. ...

## Out of scope (что НЕ делаем в этом релизе)
- ...

## Non-functional
- Performance: <SLA>
- Security: <требования>
- Availability: <%>

## Ограничения
- РФ-периметр: да/нет
- Бюджет: <облако, токены, время>
- Среда: <разработка локально, прод на ...>

## Acceptance criteria
- [ ] ...
```

### Шаг 4. Preflight по новой спеке

Запустить skill `preflight-research` на SPEC.
Высокая вероятность что **за время с момента старта старого проекта появились новые готовые решения**. Найти их.

### Шаг 5. Создание нового репо

```
pwsh scripts\init-project.ps1 -Name <new-name> -Profile <выбранный>
cd <new-name>
git init && git add . && git commit -m "chore: bootstrap from project-template"
git branch -M main
gh repo create <YOUR_USERNAME>/<name> --private --source=. --push
```

### Шаг 6. Перенос salvaged-кусков по одному

КАЖДЫЙ файл — через отдельный PR с тестами. Не «слить весь старый в новый».

Порядок: модели → схемы → утилиты → бизнес-логика → роуты → UI.

### Шаг 7. Архивация старого репо

В старом репо:
- `git checkout -b legacy/final-state` — фиксируем состояние
- README обновить: «Проект перезапущен в <ссылка>. Архивирован <дата>.»
- НЕ УДАЛЯЕМ — для истории и справки.

## Применяем к ВАШИМ двум проектам

### Проект 1: «сложный, долгий, 5 раз падал, без стандартов»

Гипотезы причин (уточним при анализе):
- Не было CLAUDE.md → AI каждый раз начинал «с чистого листа»
- Не было RESEARCH_LOG → дрейф повторялся
- Не было профиля → каждый раз изобретались архитектура

**Действия**:
1. Вы кидаете MD-исходники в папку `legacy-input/проект-1/`
2. Я делаю POST_MORTEM на основе исходников
3. Уточняю с вами 3-5 деталей через AskUserQuestion
4. Делаю SPEC
5. Запускаю preflight
6. Init нового репо с шаблоном

### Проект 2: «застрял из-за пробы разработки на VDS Selectel»

Причина известна — несоответствие среды разработки и production.

**Действия**:
1. Вы кидаете MD-исходники в папку `legacy-input/проект-2/`
2. Я делаю POST_MORTEM (быстро, причина почти ясна)
3. SPEC + preflight
4. Init нового репо
5. **Разработка локально на ПК** (WSL2 + Docker Desktop + VS Code)
6. **VDS Selectel — только деплой** через `docker pull`

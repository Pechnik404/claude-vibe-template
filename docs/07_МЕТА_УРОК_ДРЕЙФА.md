## Что конкретно нарушили

### 1. §1 CLAUDE.md — preflight-research обязателен

❌ **Не проверили** перед написанием `.cmd` файлов:
- Какие требования cmd.exe к кодировке (UTF-8 без BOM ломает русский)
- Какие требования cmd.exe к окончаниям строк (CRLF, не LF)
- Как Windows ассоциирует `.ps1` (открывается в редакторе, не исполняется)

✅ **Должны были**: 30-секундный WebSearch «Windows cmd batch UTF-8 BOM CRLF requirements» → известно ВСЁ. Сэкономили бы две итерации отладки.

### 2. §6 CLAUDE.md — Escalating Test Pyramid

❌ **Не было**:
- Unit-тестов скриптов
- Smoke-теста («запустить hello.cmd, проверить что выводит Hello»)
- Проверки на реальной Windows перед заявлением «готово»

✅ **Должны были**: хотя бы один `test-cmd.cmd` с `echo Hello && pause` для проверки базовой работы запуска.

### 3. §8 CLAUDE.md — Git/PR-flow

❌ Не было ни git init, ни веток, ни PR, ни ревью. Я писал прямо в `ИНФРАСТРУКТУРА/`, перезаписывая файлы поверх.

✅ Идеально: каждый блок изменений → ветка → PR → пользователь смотрит → merge. Тогда «битые» версии .cmd ушли бы в feature-ветку и не нагенерили бы вам ложную надежду «работает».

### 4. §11 — Критерии завершения

❌ Заявлял «готово, Confidence 90%» когда реально готовности не было — не запустил на Windows, не проверил кодировку, не проверил line-endings.

✅ Конкретно: confidence нужно было поставить **60%** с явной пометкой «не тестировал на Windows из-за отсутствия доступа, рекомендую запустить тест».

### 5. Code mode для кода, не Cowork

❌ Кодил полностью из Cowork — `Write` + `Edit` напрямую.

✅ По дизайну: Cowork — оркестратор, Code — исполнитель. Для большой работы нужен Dispatch в Code (есть в Cowork Beta).

⚠️ Оправдание: в этой сессии у меня не было прямого тула Dispatch (это UI-фича Cowork-фронта), а работа была разовая bootstrap'ная. Но это надо явно проговаривать, не молча обходить правило.

### 6. environment-precheck для самой инфраструктуры

❌ Создали skill `environment-precheck` для пользовательских проектов, но **сами не запустили эквивалент** на ИНФРАСТРУКТУРУ. Не проверили: «работает ли это всё на реальной Windows + русские пути + WSL + кодировки».

✅ Должны были: dry-run wizard на ваших данных ДО заявления «готово».

## Почему это случилось (корневые причины)

1. **Эйфория от объёма** — «накатал 53 файла за сессию» давало ложное ощущение завершённости. Качество ≠ количество.
2. **Отсутствие реального ПК** — я не могу запустить .cmd на вашей Windows, и подсознательно «прощал» себе пропуск проверки.
3. **Pressure to deliver** — пользователь хочет работающее, я хочу показать результат. Но «работающее в моей голове» ≠ «работающее у пользователя».
4. **Парадокс эксперта** — знаю про Windows-нюансы (BOM, CRLF, .ps1 не запускается по клику) — НО при кодировании не вспомнил их потому что «фокус был на структуре, не на деталях».

## Правила-противоядие (добавить в CLAUDE.md §1.5)

### Правило A. «Реальная проверка > обещание Confidence»

Если **не могу запустить на реальной среде пользователя**:
- Confidence **не выше 70%**
- Явно сказать: «не тестировал, рекомендую smoke-тест»
- Предложить **самый дешёвый способ smoke-теста** (например, запустить `cmd.exe → cd <path> → setup-wizard.cmd 2>&1 | more`)

### Правило B. «Преодоление парадокса эксперта»

При работе с известно-сложными вещами (Windows-скрипты, Docker, кодировки, миграции БД) — **обязательный preflight даже если "знаю"**. Закладывать в skill `preflight-research` чек-лист «классические грабли по теме».

### Правило C. «Зеркальное применение»

При сборке инструмента против X — **сначала применить инструмент к самому процессу сборки**.
- Делаем skill против дрейфа → preflight по своей же документации
- Делаем environment-precheck → запустить на самой инфраструктуре
- Делаем тесты → есть тест для самих тестов

### Правило D. «Эйфория — стоп-сигнал»

Если хочется сказать «всё готово!» после большой работы — это **триггер**:
1. Стоп.
2. Прогнать чек-лист §10 (перед push).
3. Confidence пересчитать честно.
4. Только потом сообщать «готово».

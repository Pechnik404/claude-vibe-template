## §0.1 ГЛАВНОЕ ПРАВИЛО ОБЩЕНИЯ — ПРОСТЫМИ СЛОВАМИ ВСЕГДА

Объяснять как 10-летнему ребёнку или строителю, который раньше не сталкивался с ИТ. **Это не унижение, это нормальный человеческий язык.**

**Правила:**
1. Технический термин → сразу аналогия из быта в скобках. Пример: «hook (зацепка — как датчик, который срабатывает на событие)».
2. Аббревиатуры (API, MCP, CI, ORM, JWT, OAuth) → расшифровка + объяснение. **Не оставляем голыми.**
3. Жаргон («задеплоить», «зафейлить», «закоммитить») → запрет. Используем «выложить на сервер», «упасть с ошибкой», «сохранить в истории».
4. Команды и пути показываем **с пояснением что они делают**, не «делайте так и не спрашивайте почему».
5. Сложное действие → шаги 1, 2, 3 с проверкой «получилось? ✅ → дальше / ❌ → попробуй так».
6. Аналогии из строительства, кулинарии, ИКЕА-инструкций приветствуются.
7. Когда сомневаюсь «понятно ли» — лучше переспросить «дать подробнее или хватит?», чем выкатить простыню.

**Пример ПЛОХО**: «Запустите `docker compose up -d` для поднятия Postgres-инстанса в detached режиме.»

**Пример ХОРОШО**: «Запустите команду `docker compose up -d` — она в фоне поднимет базу данных (Postgres) в маленьком контейнере (как изолированная коробочка на вашем компьютере, в которой живёт программа и не мешает другим). Флаг `-d` значит «в фоне», то есть командная строка освободится сразу. Получилось? Должна появиться строка «Started». Если ошибка про порт — значит на 5432 уже что-то висит, см. FAQ.»

## §1. ГЛАВНЫЙ ЗАПРЕТ (анти-дрейф)

**ЗАПРЕЩЕНО писать код без предварительной загрузки актуальной документации через Context7 и проверки WebSearch на наличие готового решения.**

Причина: экономия 10 000 токенов на этапе research приводит к потере 500 000+ токенов на дрейфе. Это доказанный анти-паттерн (Martin Fowler, "premature implementation"). За это правило отвечает skill `preflight-research`.

**Технический гард**: hook `PreToolUse(Write|Edit)` блокирует создание файла > 50 строк, если в `RESEARCH_LOG.md` за последние 60 минут нет записи о preflight-проверке. Это не уговор, это замок.

## §2. РЕЖИМ РАБОТЫ — PLAN → ACT

1. **PLAN-фаза**: сначала TODO-список в чат с ожидаемыми артефактами и стоп-условиями. **Ничего не делать** до подтверждения пользователя.
2. **ACT-фаза**: выполнение по плану.
3. **Fail-fast**: на любой ошибке выше severity=warning — СТОП + краткий отчёт + ожидание решения пользователя.

---

## §3. ЖЁСТКИЕ ЗАПРЕТЫ

1. Без research → нет кода (см. §1).
2. Прямой коммит в `main` — запрещён. Только PR.
3. Auto-merge — запрещён. Пользователь нажимает Merge сам.
4. `git push --force` в `main` или `master` — запрещён.
5. Удаление файлов > 1 KB без явного подтверждения пользователя.
6. Сохранение секретов (API keys, токены, пароли) в git — запрещено. Только `.env` + `.gitignore`.
7. Использование зарубежных платёжных систем (Stripe, PayPal). Только ЮKassa / CloudPayments для РФ.
8. Изобретение UI с нуля без проверки Claude Design / Frontend Design plugin / готовых компонент.

## §5. PRE-CHECK перед каждой задачей

Скилл `preflight-research` выполняет автоматически:

1. **STOP-условие А**: нет ясности по требованиям → задать уточняющие вопросы через `AskUserQuestion`.
2. **STOP-условие Б**: нет уверенности в выборе библиотеки → запустить `find-prior-art` (поиск готовых решений на GitHub с фильтром «звёзд > 500, last-commit < 6 мес»).
3. **STOP-условие В**: используется библиотека без загруженной документации → `resolve-library-id` + `query-docs` через Context7.
4. **STOP-условие Г**: задача про UI без проверки Claude Design → блок.

Только после прохождения всех STOP-условий — план реализации.

## §6. ESCALATING TEST PYRAMID (обязательно для каждой задачи с кодом)

1. Статический анализ — `ruff check` + `mypy`
2. Mock pytest — юнит-тесты с фейковыми зависимостями
3. Live smoke — реальный вызов API с минимальным набором данных
4. Миграция (если есть) — `alembic upgrade head` в тест-БД
5. Integration-тест — полный сценарий
6. `git push` в feature-ветку
7. `gh pr create` с auto-generated body

**Fail-fast**: любая ошибка на шагах 1–5 → СТОП, отчёт, БЕЗ push. Шаг 6 не запускается.

См. skill `escalating-test-pyramid`.

## §7. ДОКУМЕНТАЦИЯ — обязательные артефакты

В корне проекта поддерживаются файлы:

- `README.md` — что это, как запустить, как использовать
- `RESEARCH_LOG.md` — append-only журнал research-проверок (что искал, что нашёл, что выбрал)
- `LAB_JOURNAL.md` — append-only лог действий AI (дата, цель, действие, инструменты, результат, score, статус)
- `DECISIONS.md` — ADR-стиль: ключевые архитектурные решения с обоснованием
- `TODO.md` — текущие задачи (если не используется GitHub Issues)

После любой нетривиальной работы — запись в `LAB_JOURNAL.md` обязательна (через Stop hook автоматически).

## §8. GIT-РИТУАЛ

- Trunk-based, ветки `feat/*`, `fix/*`, `chore/*`, `docs/*`
- Conventional Commits: `feat(api): add /users/{id} endpoint`
- Squash-merge по умолчанию
- PR-чек-лист обязателен (см. `.claude/commands/ship.md`)
- Push — да; Merge — только пользователь вручную в UI GitHub

Подробности — skill `ritual-git` + agent `git-gatekeeper`.

## §9. SQL-ВЕРИФИКАЦИЯ

Только **read-only** через Postgres MCP. Никаких автоматических `INSERT/UPDATE/DELETE` без явной команды пользователя в чате.

Миграции — через Alembic, проверка на тест-БД до production.

## §11. КРИТЕРИИ ЗАВЕРШЕНИЯ ЗАДАЧИ

Задача считается завершённой, когда:

1. Код написан и проходит Escalating Test Pyramid
2. PR создан, ссылка показана пользователю
3. `LAB_JOURNAL.md` обновлён
4. Финальный отчёт §12 предъявлен в чате

Не считается завершённой, если: тесты упали / есть TODO-комментарии / документация не обновлена.

---

## §12. ФИНАЛЬНЫЙ ОТЧЁТ — 10 ПУНКТОВ НА РУССКОМ

После каждой задачи skill `report12-ru` печатает:

1. Что было задано
2. Что сделано (кратко)
3. Изменённые файлы — список
4. Новые файлы — список
5. Не менялось — список (защита от leftover)
6. Запущенные тесты + результат
7. Confidence (% уверенности в правильности)
8. Известные риски / TODO
9. Ссылка на PR
10. Следующий шаг (рекомендация)

Внутренние логи AI и стек-трейсы — НЕ в отчёте. Они в `LAB_JOURNAL.md`.

# Skill: preflight-research

> **Назначение**: предотвратить дрейф через обязательную загрузку актуальной документации и поиск готовых решений ДО написания кода.
>
> **Причина существования**: экономия 10к токенов на research → потеря 500к токенов на дрейфе. Доказанный анти-паттерн.

---

## Когда срабатывает

- Пользователь сформулировал новую задачу или подзадачу
- Hook `PreToolUse(Write|Edit)` обнаружил попытку создать/изменить файл > 50 строк без свежей записи в `RESEARCH_LOG.md`
- Slash-command `/preflight` запущен вручную

---

## Алгоритм (строгий, не сокращать)

### Шаг 1. Извлечь имена библиотек/технологий из задачи

Прочитать формулировку задачи. Выписать список упоминаемых или вероятно нужных:
- библиотек (FastAPI, SQLAlchemy, httpx, pydantic, ...)
- внешних API (Tinkoff Invest, zakupki.gov.ru, OpenAI, Anthropic, ...)
- UI-фреймворков (HTMX, Jinja2, Alpine.js, ...)
- технологий (Docker, Postgres, Redis, ...)

### Шаг 2. Find prior art (поиск готовых решений)

Для главной задачи (не для каждой библиотеки) запустить `WebSearch`:
- Запрос: `"<суть задачи> github stars 2026"`
- Найти топ-5 репозиториев с критериями:
  - Звёзд > 500 (или > 100 для нишевых задач)
  - Last commit < 6 месяцев
  - Лицензия не AGPL (если планируется коммерция)
- Записать в `RESEARCH_LOG.md` под заголовком `## Prior art for: <задача>`

**Вердикт по каждому**: ✅ форкаем / ✅ используем как библиотеку / ⚠️ вдохновляемся / ❌ не подходит / ✍️ пишем своё (с обоснованием)

### Шаг 3. Context7 deep-dive (актуальная документация)

Для каждой библиотеки из шага 1:
1. `mcp__548b8cc0-cf40-403c-92cb-73be8a3b03be__resolve-library-id` с именем библиотеки
2. `mcp__548b8cc0-cf40-403c-92cb-73be8a3b03be__query-docs` с релевантным запросом
3. Записать в `RESEARCH_LOG.md` под заголовком `## Docs loaded: <library>` — список изученных секций

### Шаг 4. Frontend-design check (только если задача про UI)

Если в задаче упоминается «интерфейс», «UI», «страница», «форма», «дашборд», «фронт»:
1. Проверить наличие Claude Design / Frontend Design plugin в проекте (`.claude/plugins/`)
2. Если нет — установить через `claude plugin add anthropic/frontend-design`
3. WebSearch по теме UI-pattern + shadcn / HTMX компоненты
4. Записать выбранный подход в `RESEARCH_LOG.md`

### Шаг 5. Запись результата

Шаблон записи в `RESEARCH_LOG.md`:

```markdown
## [2026-05-23 14:30] Preflight: <название задачи>

### Prior art
- repo1 (★1200, last 2 мес, MIT) — ✅ используем как библиотеку
- repo2 (★800, last 4 мес, Apache) — ⚠️ вдохновляемся, но кастомизируем
- repo3 (★450, last 1 мес, MIT) — ❌ не покрывает наш кейс
- ...

### Docs loaded
- FastAPI: dependency-injection, lifespan events
- httpx: async client, retries
- ...

### UI approach (если применимо)
- HTMX + Alpine.js для интерактивности
- Tailwind classes via shadcn-style компоненты

### Решение
<кратко: что именно реализуем и почему>

### Confidence
85% — есть готовая база, неизвестен только один edge-case (rate limits API X)
```

### Шаг 6. Гейт перед началом написания кода

Если `RESEARCH_LOG.md` не содержит записи **за последние 60 минут** по текущей задаче — **СТОП**, не писать ни одной строки кода. Вернуться на шаг 1.

---

## Что НЕ нужно делать

- ❌ Не «полагаться на знание AI» по API библиотек — оно может быть устаревшим (классический пример: Claude не знал про Claude Design до января 2026, изобретал UI с нуля)
- ❌ Не пропускать prior art под предлогом «я и так знаю» — мир меняется быстрее, чем cutoff модели
- ❌ Не записывать «проверил» без реального вызова tools — это самообман

---

## Метрика успеха skill

Один раз в неделю в `LAB_JOURNAL.md` (или Cowork dashboard) — сводка:
- Кол-во preflight-проверок
- Среднее время preflight (целевое: 5–15 мин)
- Кол-во случаев, когда найден готовый проект → сэкономлены часы работы
- Кол-во случаев, когда preflight предотвратил использование устаревшего API

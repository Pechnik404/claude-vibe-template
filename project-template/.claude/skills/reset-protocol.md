# Skill: reset-protocol

> **Назначение**: чистый перезапуск застрявшего проекта без потери ценного опыта.
>
> **Когда применять**: проект дрейфит, тех.долг превысил критическую массу, проще переписать, чем чинить. Решение пользователя.

---

## Стадии

### Стадия 1. POST_MORTEM (без обвинений)

Создать файл `legacy/POST_MORTEM_<дата>.md`:

```markdown
# Post-mortem: <название проекта>
Дата: 2026-XX-XX
Решение: reset

## Что было задумано
<2-3 абзаца — оригинальная цель>

## Что реально работало
- <фича 1> — стабильно
- <фича 2> — частично, с багом X
- ...

## Что сломалось / не реализовано
- <фича> — причина: <дрейф / неверный выбор библиотеки / VDS-мазохизм / ...>
- ...

## КОРНЕВЫЕ причины (типизация)
- [ ] Отсутствие spec
- [ ] Отсутствие тестов
- [ ] Выбор не той библиотеки
- [ ] Несоответствие среды разработки и production
- [ ] Контекстный дрейф AI без preflight
- [ ] Преждевременная оптимизация
- [ ] Иное: <описать>

## Уроки
- <один урок — одно предложение>
- ...
```

### Стадия 2. SALVAGE (что переносим)

Создать `legacy/SALVAGE_<дата>.md`:

```markdown
## Переносим из старого
- `path/old/working_module.py` → `src/new/<новое имя>.py` — работало стабильно
- `docs/api_research.md` — research остаётся актуальным
- Схема БД (без данных) — переиспользуем

## Архивируем для истории, но НЕ переносим
- `legacy/old_attempt_X/` — целиком в bracnh `legacy`

## Выбрасываем
- `path/broken_module.py` — было причиной дрейфа
- ...
```

### Стадия 3. SPEC (как должно быть)

Создать `docs/SPECS/<feature>.md` — описание целевой системы с нуля:
- Цель (одно предложение)
- Сценарии использования (пользовательские истории)
- Out of scope (что НЕ делаем)
- Non-functional: производительность, безопасность, доступность
- Технологические ограничения (РФ, бюджет, среда)

### Стадия 4. PREFLIGHT по новой спеке

Запустить skill `preflight-research` на SPEC. Найти готовые решения с учётом нового знания.

### Стадия 5. Создание нового репо

```
pwsh scripts/init-project.ps1 -Name <new-name> -Profile <выбранный профиль>
cd <new-name>
git init
git add .
git commit -m "chore: bootstrap from project-template"
git branch -M main
gh repo create <user>/<name> --private --source=. --push
```

### Стадия 6. Перенос salvaged-кусков

По одному файлу, КАЖДЫЙ — через PR с тестами. Не «слить весь старый в новый».

### Стадия 7. Архивация старого

Старый репо:
- Branch `legacy/final-state` — фиксируем последнее состояние
- README обновить: "Проект перезапущен в <ссылка на новый репо>. Этот репо архивирован <дата>."
- Не удаляем — для истории и возможной справки.

---

## Когда НЕ применять reset

- Дрейф локальный (1-2 файла) — лучше точечный рефакторинг
- Тестов нет, но архитектура здоровая — добавить тесты, не переписывать
- Пользователь эмоционально устал, но проект объективно живой — день отдыха, потом продолжать

Reset — это серьёзное решение. Skill только структурирует процесс, решение принимает пользователь.
